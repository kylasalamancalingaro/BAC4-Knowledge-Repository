Framework Formulation for Standard vs Bespoke Data Products

A workshop-first approach for defining the standards that separate reusable, scalable data products from one-off bespoke delivery.

Page 1: core standards workshop Standard architecture + practices Decision baseline for all later steps

Core premise

Step 1 is the crucial step and should be treated as the foundation of the whole effort. The outcome is a shared standards baseline that later steps operationalize.

If the standard is unclear, every downstream prioritization, discovery, and ADO delivery decision becomes unstable.
1

Identify Standards — Subject of Workshop

Define what “standard” means before selecting, scoring, or delivering products.

Definition of “standard”

Agree the reusable components and boundaries that qualify a product as standard.

  • Shared Lakehouse?
  • Shared semantic model?
  • Shared reporting layout?
  • Reference Mike’s standards definition document, if available.

High-level standard architecture

Use the architecture baseline, including Sol Arch's diagram, to anchor technical consistency.

  • Source to ingestion path
  • Lakehouse / curated layer
  • Semantic and consumption layer

Domains covered

Identify which business or analytical domains are in scope for standardization.

  • Finance, CSAT, supply, sales, HR, etc.
  • Cross-client applicability
  • Domain ownership expectations

Data product types

Clarify which deliverable forms can be standardized.

  • Power BI report
  • Output table
  • CSV or extract
  • Reusable semantic model

Potential standard products

Create the candidate list that can later be shortlisted into base products.

  • Known repeatable reporting needs
  • Existing bespoke products with reuse potential
  • Domain-level candidate inventory

Standard processes and practices

Document how standard delivery differs from bespoke delivery.

  • Process flows
  • CI/CD expectations
  • End-to-end solution management
  • Governance and support model

Operationalize the Identified Standards

Steps 2–5 translate the Step 1 standards baseline into shortlist selection, prioritization, discovery iteration, and ADO delivery structure.

Flexible after Step 1 DVF scoring Discovery → ADO pipeline

How to use this page

This page is intentionally separate from the core standards workshop. The operating approach may change depending on the standards agreed in Step 1.

Treat Steps 2–5 as the execution model, not the definition of the standard itself.

From Candidate Standards to Delivery Pipeline

2

Identify possible base products

Use Step 1e as the shortlist and select the best base product per domain from the existing list.

3

Prioritize standard products

Apply DVF scoring to rank candidates by desirability, viability, and feasibility.

4

Discovery for standard products

Iterate from the selected base products and identify feature, process, practice, and architecture changes.

5

Standard products in ADO / Delivery pipeline

Decide what percentage of sprint capacity supports long-term standard-product goals versus bespoke demand.

Step 3: DVF Prioritization

Score AreaWhat to assess
DesirabilityHow many clients or domains need it? Is the business problem recurring?
ViabilityCan it be owned, governed, supported, and evolved as a standard product?
FeasibilityCan the architecture, data availability, CI/CD, and delivery model support it?

Step 4: Discovery Inputs

InputRole in discovery
Best judgementUseful, but not ideal if used alone. Should be validated against evidence.
Similar bespoke reportsCompare and combine existing products to identify repeatable commonalities.
2+ client requirementsUse multiple-client needs to define high-level requirements for the standard product.

Step 5: ADO / Delivery Pipeline Structure

Create EPIC

Represent the long-term standard-product objective and link it to the agreed Step 1 standards baseline.

Create Features

Break the standard product into major capability areas, domain versions, architecture enablers, or process improvements.

Create User Stories

Translate discovery outcomes into sprint-ready backlog items with acceptance criteria and delivery evidence.

Capacity principleExplicitly allocate sprint capacity between standard-product evolution and bespoke delivery.
Traceability principleEvery delivery item should trace back to a base product, DVF rationale, and discovery finding.
Iteration principleStandard products evolve through repeated discovery and implementation cycles, not one-time design.

Overall Decision Flow

PhasePrimary questionOutput
1. Identify standardsWhat makes a data product standard instead of bespoke?Standards definition, architecture, domains, product types, candidate products, standard practices.
2. Identify base productsWhich existing product is the best starting point per domain?Base-product shortlist.
3. PrioritizeWhich standard products should be pursued first?DVF-ranked standard-product backlog.
4. DiscoverWhat needs to change to make the base product a true standard?Feature, process, practice, and architecture change set.
5. Deliver in ADOHow do we fund and manage standard-product work over time?EPIC, Features, User Stories, and capacity allocation model.
